1) Getting Started: Install, Run, and Verify

This project is a Python + pyglet voxel game. If you are brand new, follow this exact checklist in order.

  1. Open a terminal in the repository root.
  2. Create a Python virtual environment.
  3. Activate it and install dependencies.
  4. Run the game and confirm a window appears with terrain + crosshair.
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python main.py

Alternative launcher (same game behavior):

python minecraft_clone.py

Run with explicit deterministic terrain seed:

python main.py --seed 12345

Optional experimental meshing mode (may be faster, may produce UV artifacts on some GPUs/drivers):

python main.py --use-texture-array

What "success" looks like

  • Window opens at 1280×720.
  • You first see a loading screen while nearby chunks are generated/meshed.
  • Then terrain renders, crosshair appears, and a 9-slot hotbar is visible.
  • Mouse click captures pointer; WASD movement and block interactions work.

3) Runtime Lifecycle: From Process Start to Shutdown

Startup sequence

  1. main.py parses --seed and --use-texture-array.
  2. GameWindow is created.
  3. setup_gl() configures clear color + alpha blending.
  4. pyglet app loop starts.

GameWindow initialization highlights

  • Creates profiler and clears old lag reports.
  • Builds a World with terrain seed and renderer mode.
  • Primes a tiny chunk area synchronously, then requests visible chunks.
  • Spawns player at height_at(0,0) + 2.
  • Configures a scheduled update callback at TICKS_PER_SECOND (60).
  • Builds UI elements (crosshair, text label, hotbar rectangles/labels).
  • Enters temporary loading-screen mode until required chunks are fully rendered.

Per-frame update model

  • Update time is clamped (dt ≤ 0.25) to avoid huge physics jumps.
  • Update may be split into substeps (targeting 60Hz simulation granularity).
  • Applies movement vector from key states.
  • Applies gravity and terminal velocity limits.
  • Resolves collisions against solid blocks.
  • Refreshes visible/active chunks on an adaptive stream timer.

Per-frame draw model

  • Draw mode A: atlas debug view (I key).
  • Draw mode B: loading screen (until chunk render readiness).
  • Draw mode C: normal game render → 3D world first, 2D UI second.

Shutdown

  • Profiler writes timestamped and latest reports (.txt and .json).
  • World executors (chunk + mesh workers) are shut down cleanly.

4) Gameplay Mechanics (Implemented vs Not Implemented)

Implemented mechanics

  • First-person look + movement (WASD + mouse).
  • Gravity + jump + collision against solid blocks.
  • Block break (left click) using ray cast target selection.
  • Block placement (right click) from selected hotbar stack.
  • Inventory hotbar with 9 slots and stack size 64.
  • Procedural terrain generated from seed.
  • Chunk streaming and distance-based unloading.

Not implemented in current runtime code

  • No active fly mode toggle in input handling.
  • No active crafting system implementation in gameplay loop.
  • No active mob entity simulation module in source tree.

Important for students: Some older README wording references features that are not present in current gameplay code. Use source files as the ground truth when extending behavior.

Example interaction flow: Breaking a block

  1. Left click in exclusive mouse mode.
  2. world.hit_test(position, view_vector) finds target block + previous empty cell.
  3. world.remove_block(target) validates breakability and removes.
  4. Removed block type is added to inventory via inventory.add().
  5. Affected chunk meshes are rebuilt so visuals update immediately.

5) World & Chunk System: Deep Dive

Core world constants

  • Chunk size: 16×16 columns in X/Z (SECTOR_SIZE).
  • World height: 256 Y levels.
  • Visible radius: 4 chunks from center.
  • Active radius: 5 chunks desired for loading logic.
  • Unload radius: 6 chunks.
  • Loaded chunk soft cap: 140 chunks.

Data structures you must understand

  • blocks: global map (x,y,z) -> block_name for currently loaded chunks.
  • _chunk_blocks: reverse index chunk -> set(positions).
  • _dirty_chunks: chunks needing mesh rebuild.
  • _loaded_chunks / _desired_chunks / _visible_chunks: state sets for streaming policy.
  • _chunk_cache: recently unloaded block data to avoid full regeneration when revisiting.
  • _modified_blocks: persistent per-position overrides from player edits.

Chunk generation path

  1. Determine center chunk from player X/Z.
  2. Compute desired and visible chunk sets.
  3. Queue missing chunks to thread pool.
  4. When a future completes, apply generated data on main thread.
  5. Mark chunk dirty and queue mesh build.
  6. Upload mesh on main thread and store render handle.

Terrain composition by Y level

  • Y=0: bedrock.
  • Y=height: grass.
  • Y in height-2..height-1: dirt.
  • Lower than that: stone.

Why border chunks are special

When a chunk loads/unloads, adjacent chunk meshes can become visually stale at the seam. The world marks neighbor chunks dirty so border faces rebuild and stay consistent.

6) Rendering Pipeline: Cameras, Meshes, and Textures

High-level rendering layers

  1. Projection setup: set_3d() builds perspective matrix for world draw.
  2. Culling: back-face culling enabled during world rendering.
  3. Batch draw: all chunk mesh parts render via pyglet batch.
  4. UI pass: switch to orthographic with set_2d() and draw hotbar/text/crosshair.

How chunk meshes are efficient

  • Renderer checks block neighbors and emits only exposed faces.
  • Hidden interior faces are omitted (major triangle count reduction).
  • Chunk meshing is done on worker threads as pure data build.
  • GPU upload happens on main thread for rendering safety.

Texture behavior

  • Block definitions can specify all-faces texture or per-face texture overrides.
  • Atlas mode supports debug visualization (toggle with I).
  • Texture-array mode is optional and experimental via CLI flag.
  • If face texture file is missing, renderer falls back to block color.

7) Block System: Authoring Block Definitions Correctly

Every block is loaded from a text file in Blocks/. One file = one block type.

Supported keys in block text files

KeyTypeMeaningExample
namestringBlock ID used in world/inventory.name=grass
solidboolWhether collisions treat this block as solid.solid=true
breakableboolWhether player can remove with left click.breakable=false
colorr,g,bFallback display color (0..1 or 0..255 values).color=106,170,64
texturefilenameDefault texture for all faces.texture=stone.png
texture_topfilenameOverrides top face texture.texture_top=grass_top.png
texture_bottomfilenameOverrides bottom face texture.texture_bottom=dirt.png
texture_sidefilenameOverrides side faces.texture_side=grass.png

Example: custom block file

# Blocks/marble.txt
name=marble
solid=true
breakable=true
color=220,220,230
texture=marble.png

Then add Blocks/Textures/marble.png, restart game, and the block type becomes loadable in code.

8) Inventory System: Hotbar Rules and Stack Logic

  • Hotbar has fixed 9 slots.
  • Each slot can hold one block type up to stack size 64.
  • add(block, count) first fills partial stacks, then empty slots.
  • remove_selected(count) only consumes from active slot and returns success/failure.
  • selected_block() and selected_count() drive HUD label + placement logic.

Practical consequence

If placement fails unexpectedly, verify the selected slot is not empty and has at least one item.

10) Modification Guides (Step-by-Step for Students)

Guide A: Add a new block type

  1. Create Blocks/my_block.txt with required fields.
  2. Create texture PNG in Blocks/Textures/.
  3. Restart game (block registry loads at import time).
  4. Optional: add block into world generation logic if you want natural spawning.

Guide B: Change movement feel

  1. Open engine/constants.py.
  2. Adjust WALK_SPEED, GRAVITY, MAX_JUMP_HEIGHT.
  3. Rerun game and test movement/collision.

Guide C: Change chunk streaming radius

  1. Edit VISIBLE_RADIUS_CHUNKS, ACTIVE_RADIUS_CHUNKS, UNLOAD_RADIUS_CHUNKS in World.
  2. Increase cautiously: larger radius means more CPU + memory + mesh work.
  3. Use profiler reports to validate impact.

Guide D: Add a new keybinding action

  1. Implement branch in GameWindow.on_key_press().
  2. If stateful, add instance variable in __init__.
  3. Integrate behavior into update() and/or on_draw().
  4. Document control in this docs page and README.

11) Performance & Profiling: How to Diagnose Lag

Built-in profiler

  • Profiler tracks frame durations and named section durations.
  • Slow frames over threshold are stored with context and section breakdown.
  • Reports are written on close: timestamped + latest aliases.

Report files

  • profiling/lag_report_latest.txt (human readable summary)
  • profiling/lag_report_latest.json (machine-friendly stats)

Tuning levers in world streaming

  • Chunk request/apply per update limits.
  • Mesh rebuild queue limits.
  • Main-thread upload budgets for chunks and meshes.
  • Hard cap unload pacing to avoid long frame stalls.

General rule: prioritize smooth frame pacing over max throughput when editing these values.

12) Debugging & Troubleshooting

Window opens but appears black.

Likely OpenGL context/driver issue. Confirm pyglet can create GL context and that your system supports required GL features.

Blocks are invisible or textures look wrong.

Check that texture filenames in block definitions exactly match files in Blocks/Textures/. Try disabling --use-texture-array to compare behavior.

Cannot place blocks.

Selected hotbar slot may be empty. Break blocks first, scroll/select slot with items, then place.

Chunk streaming feels delayed.

This is expected to some degree because loading/meshing is intentionally budgeted to protect frame-time consistency.

I changed a Blocks/*.txt file but behavior didn't change.

Block registry is loaded at import time; restart the process after edits.

13) Code Reading Roadmap (For Total Beginners)

  1. Start: main.py + minecraft_clone.py (startup only).
  2. Then: engine/constants.py (gameplay knobs).
  3. Then: engine/game/window.py (input, update loop, UI).
  4. Then: engine/gameplay/inventory.py (simple, self-contained logic).
  5. Then: engine/blocks/registry.py + sample block files.
  6. Then: engine/world/terrain.py (math behind height map).
  7. Then: engine/world/world.py (largest systems integration module).
  8. Finally: renderer internals + profiler reporting.

Study tip: when reading world.py, annotate each set/dict with "ownership" and "lifecycle" notes. Most complexity is state transitions, not math.

14) FAQ

Q: Is world generation repeatable?
A: Yes. Same seed gives same base terrain heights and block layers.

Q: Are player edits persistent while chunks unload/reload?
A: Yes during runtime. _modified_blocks and per-chunk modified-position tracking re-apply edits when a chunk is reloaded.

Q: Why are there two launch files?
A: main.py includes CLI options; minecraft_clone.py is a convenience alias.

Q: Can I edit docs safely without touching game code?
A: Yes. Documentation is standalone static files under Documentation/.

15) Glossary

  • Voxel: 3D discrete cell (here, a block).
  • Chunk: Group of block columns managed as streaming/rendering unit.
  • Frustum culling: Limiting work for chunks likely outside view direction.
  • Mesh: Vertex data submitted to GPU for drawing geometry.
  • Dirty chunk: Chunk flagged for mesh rebuild because blocks changed or neighbor borders changed.
  • Hit test / ray cast: Line-of-sight traversal used to find targeted block.
  • Main-thread upload: Rendering object creation done on render thread for safety.
  • Seed: Number initializing deterministic procedural generation.
